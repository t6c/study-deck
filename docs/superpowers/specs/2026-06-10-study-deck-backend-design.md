# Study Deck Backend Design

## Purpose

Build the backend API for a Knowt-like study app with Anki-style spaced repetition. The MVP is a Spring Boot API only; frontend architecture is intentionally out of scope.

The root Spring Boot project is the production codebase. `StudyDeckManager/` is reference-only old Hibernate console code and should not be committed as part of the real application.

## Scope

Included in MVP:

- Single-user/global data model.
- Folder organization for decks.
- Deck CRUD.
- Flashcard CRUD.
- Text flashcards with optional term and definition image URL fields.
- Starred flashcards.
- Flashcard viewer and Knowt-like card sorting mode.
- Learn mode with persisted session progress.
- Practice test generation and saved submitted results.
- Matching mode generation and saved final results.
- FSRS-based spaced repetition with persisted card state and review logs.
- OpenAPI/Swagger documentation.

Excluded from MVP:

- User accounts and ownership.
- XP, streaks, rewards, or gamification.
- Audio settings, background music, sound effects, autoplay audio.
- Upload, storage, or image processing service.
- Plugin support.
- Frontend implementation.
- FSRS parameter optimization.

## Architecture

Use a layered modular monolith under `org.fpt.studydeck`.

```text
org.fpt.studydeck
  config
  controller
  dto
  domain
  repository
  service
  exception
```

Feature boundaries live inside the layers:

```text
controller.deck
controller.learn
controller.matching
controller.practice
controller.srs

service.deck
service.learn
service.matching
service.practice
service.srs

domain.deck
domain.learn
domain.matching
domain.practice
domain.srs

dto.deck
dto.learn
dto.matching
dto.practice
dto.srs
```

Controllers handle HTTP, validation, and OpenAPI annotations. Services hold business rules and session transitions. Repositories are Spring Data JPA interfaces. Entities stay inside `domain` and are not returned directly from controllers. DTOs define all request and response contracts.

## Technology

- Java 17.
- Spring Boot 3.5.x.
- Spring Web.
- Spring Data JPA.
- Spring Validation.
- Spring Security configured permissively for the MVP API and Swagger.
- springdoc-openapi for Swagger UI and OpenAPI JSON.
- MySQL runtime database support.
- H2 for local/test profile support.
- `io.github.open-spaced-repetition:fsrs` for FSRS scheduling.

FSRS is the selected scheduler because it is Anki's modern spaced-repetition direction. The MVP uses fixed default FSRS parameters and does not include per-user parameter optimization.

References:

- Anki manual, FSRS deck options: <https://docs.ankiweb.net/deck-options.html>
- Java FSRS library: <https://github.com/open-spaced-repetition/java-fsrs>

## Data Model

### Core Content

```text
folders
  id
  name
  description
  position
  created_at
  updated_at

decks
  id
  folder_id nullable
  title
  description
  visibility
  created_at
  updated_at

flashcards
  id
  deck_id
  term
  definition
  term_image_url
  definition_image_url
  starred
  position
  created_at
  updated_at
```

`decks.folder_id` is nullable because folders organize decks but do not own them. A deck can exist outside a folder.

### Spaced Repetition

```text
srs_card_states
  id
  flashcard_id unique
  fsrs_card_json
  state
  due_at
  stability
  difficulty
  scheduled_days
  elapsed_days
  reps
  lapses
  last_reviewed_at
  created_at
  updated_at

srs_review_logs
  id
  flashcard_id
  rating
  reviewed_at
  duration_ms
  previous_state
  next_state
  due_before
  due_after
  review_log_json
```

The FSRS card state is stored both as queryable columns and as `fsrs_card_json`. Queryable columns support due-card queries and stats. JSON preserves compatibility with the Java FSRS library object shape.

Supported SRS review ratings:

```text
AGAIN
HARD
GOOD
EASY
```

### Learn

```text
learn_sessions
```

Learn mode persists progress so sessions can resume and the backend owns progression rules.

### Matching

```text
matching_sessions
```

Matching sessions are generated by the backend and save final completion state. The frontend can still handle drag/drop or click interaction details.

### Practice Test

Practice tests are generated from deck cards and save submitted answers plus scoring results.

## API Design

All endpoints use base path `/api/v1`.

### Folders

```text
GET    /folders
POST   /folders
GET    /folders/{folderId}
PATCH  /folders/{folderId}
DELETE /folders/{folderId}
```

### Decks

```text
GET    /decks
POST   /decks
GET    /decks/{deckId}
PATCH  /decks/{deckId}
DELETE /decks/{deckId}

POST   /folders/{folderId}/decks/{deckId}
DELETE /folders/{folderId}/decks/{deckId}
```

### Flashcards

```text
GET    /decks/{deckId}/flashcards
POST   /decks/{deckId}/flashcards
PATCH  /flashcards/{flashcardId}
DELETE /flashcards/{flashcardId}
PATCH  /flashcards/{flashcardId}/star
```

### Deck Summary

```text
GET /decks/{deckId}/summary
```

Returns total cards, starred cards, due SRS cards, new cards, learning cards, review cards, and available mode flags.

### Flashcard Viewer And Sorting

```text
GET  /decks/{deckId}/viewer-cards?sort=original|shuffle|starred&mode=view|sorting
POST /decks/{deckId}/sorting-sessions
GET  /sorting-sessions/{sessionId}
POST /sorting-sessions/{sessionId}/answers
```

Sorting answers track Knowt-like "know" and "do not know" decisions without using the SRS scheduler.

### Learn

```text
POST /decks/{deckId}/learn-sessions
GET  /learn-sessions/{sessionId}
POST /learn-sessions/{sessionId}/answers
POST /learn-sessions/{sessionId}/complete
```

Learn questions support flashcard, multiple choice, written, and true/false question types. MVP smart grading can start with normalized exact matching and be expanded later.

### Spaced Repetition

```text
GET  /decks/{deckId}/srs/due
POST /decks/{deckId}/srs/review-sessions
GET  /srs/review-sessions/{sessionId}
POST /srs/review-sessions/{sessionId}/reviews
GET  /flashcards/{flashcardId}/srs-state
GET  /decks/{deckId}/srs/stats
```

Review submissions update the flashcard's FSRS state and append an immutable review log.

### Matching

```text
POST /decks/{deckId}/matching-sessions
GET  /matching-sessions/{sessionId}
POST /matching-sessions/{sessionId}/matches
POST /matching-sessions/{sessionId}/complete
```

Matching session creation accepts card count and starred-only options.

### Practice Test

```text
POST /decks/{deckId}/practice-tests
GET  /practice-tests/{practiceTestId}
POST /practice-tests/{practiceTestId}/answers
POST /practice-tests/{practiceTestId}/submit
```

Practice test creation accepts question type toggles, prompt side options, starred-only, shuffle, and question count.

## Error Handling

Use a global exception handler that returns a consistent JSON body:

```json
{
  "timestamp": "2026-06-10T00:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Validation failed.",
  "path": "/api/v1/decks",
  "fieldErrors": [
    {
      "field": "title",
      "message": "Title is required."
    }
  ]
}
```

HTTP status rules:

- `400` for validation failures, invalid ratings, invalid session transitions, and not enough cards for a selected mode.
- `404` for missing folder, deck, flashcard, or session.
- `409` for conflicting state, such as answering a completed session.
- `204` for successful deletes.

## Configuration

Application config should:

- Set `spring.jpa.open-in-view=false`.
- Provide H2 defaults for local/test runs.
- Allow MySQL configuration through environment variables.
- Configure Swagger/OpenAPI metadata.
- Permit `/api/**`, `/v3/api-docs/**`, `/swagger-ui/**`, and `/swagger-ui.html` without authentication for MVP.

## Testing Strategy

Use test-driven development for implementation.

Service tests:

- Folder, deck, and flashcard business rules.
- Starred filtering.
- Learn session progression.
- Matching session creation and completion.
- Practice test scoring.
- FSRS adapter behavior for `AGAIN`, `HARD`, `GOOD`, and `EASY`.

Repository tests:

- JPA relationship mappings.
- Deck-with-cards queries.
- Due-card queries by deck.

Controller tests:

- Request validation.
- Success status codes.
- Error response body shape.
- DTO serialization.

Context test:

- Full Spring context starts with MVP security and OpenAPI configuration.

## Implementation Order

1. Project configuration, security, OpenAPI, global errors.
2. Core folder/deck/flashcard entities, repositories, services, and controllers.
3. Deck summary and flashcard viewer/sorting.
4. FSRS dependency, adapter, card state, review logs, and SRS endpoints.
5. Learn sessions.
6. Matching sessions.
7. Practice tests.
8. Final OpenAPI review and integration test pass.

Each implementation slice should be independently testable and should not depend on frontend work.
